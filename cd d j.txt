To run your programming-game Spring Boot project in a secure cloud sandbox, you can configure and run everything—including the Docker builds and Jenkins automation—directly inside **GitHub Codespaces**.

By using the official **GitHub CLI (`gh`)**, you can spin up, configure, and operate your remote Codespace from your local System Command Prompt (CMD).

Since running Jenkins *inside* your game's Dockerfile violates basic container design principles (which dictates one service per container), this lesson uses the industry-standard architecture: **one container for your Spring Boot game engine** and **one separate container for Jenkins**. We will link them together inside the Codespace.

---

## 🏗️ The Cloud Architecture

Inside your GitHub Codespace (which is a secure Linux Virtual Machine in the cloud):

* **Java and Maven** will compile your source code into a JAR.
* **The Docker Daemon** running in the Codespace will build your Spring Boot image.
* **A Docker-in-Docker bridge** will spin up your Jenkins automation engine as a neighboring container.

---

## 🛠️ Step-by-Step Practical Blueprint

### Step 1: Install and Authenticate the GitHub CLI (`gh`)

Open your local system **Command Prompt (CMD)** and authenticate your GitHub account:

```cmd
gh auth login

```

*Follow the interactive prompts: Choose **GitHub.com** $\rightarrow$ **HTTPS** $\rightarrow$ **Log in with a web browser**. Copy the 8-digit code, paste it into the browser window that opens, and authorize.*

---

### Step 2: Create and Launch Your Codespace

Create and launch a cloud development workspace directly from your command prompt using your repository name (replace with your repository's path):

```cmd
gh codespace create --repo your-username/programming-game-spring --machine basicLinux32gb

```

Once the workspace is created, connect your CMD terminal directly to your cloud instance:

```cmd
gh codespace ssh -c your-codespace-name

```

*You are now logged into your cloud-hosted terminal. Every command from here down is executed inside GitHub's cloud environment.*

---

### Step 3: Organize Your Project Files

Let's verify that your project has the clean folder structure required for packaging. Run this in your Codespace shell to see your directory layout:

```bash
ls -la

```

If your `Dockerfile` is inside a subdirectory or has an extension (like `Dockerfile.yml`), rename it and bring it to your root directory so the Docker builder can discover it easily:

```bash
mv Dockerfile_folder/Dockerfile.yml ./Dockerfile

```

---

### Step 4: Configure the Game Dockerfile

Open your `Dockerfile` using the built-in command line editor:

```bash
nano Dockerfile

```

Paste this clean, multi-stage production build configuration:

```dockerfile
# Stage 1: Build the Java application
FROM maven:3.8.5-openjdk-17 AS builder
WORKDIR /app
COPY . .
RUN mvn clean package -DskipTests

# Stage 2: Minimal runtime image
FROM eclipse-temurin:17-jre-slim
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

```

*Press `Ctrl + O` then `Enter` to save, and `Ctrl + X` to exit Nano.*

---

### Step 5: Run Your First Docker Build in the Cloud

Codespaces come with the Docker daemon pre-configured. Build your programming-game image locally in the cloud:

```bash
docker build -t game-engine:latest .

```

Verify the build succeeded and the image is registered in your Codespace's local registry:

```bash
docker images

```

---

### Step 6: Deploy Jenkins alongside the App Engine

To prevent Jenkins from breaking your Game's dependencies, run Jenkins in its own container right next to your application. Run this command to launch the Jenkins master:

```bash
docker run -d -p 8082:8080 -p 50000:50000 \
  -v jenkins_home:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  --name jenkins-master jenkins/jenkins:lts

```

> **What the `-v` flags do:** We are mounting your Codespace's `/var/run/docker.sock` directly into Jenkins. This grants Jenkins the authority to build and run sibling Docker containers inside the Codespace.

---

### Step 7: Configure Permissions and Retrieve Security Key

To allow the Jenkins user to communicate with the mounted Docker daemon socket, adjust its file permissions:

```bash
docker exec -u 0 -it jenkins-master chmod 666 /var/run/docker.sock

```

Now, grab your unique automated setup security key from the logs:

```bash
docker logs jenkins-master

```

*Copy the 32-character string from the console output.*

---

### Step 8: Access Your Web Interfaces (Port Forwarding)

Because Codespaces run inside a secure cloud sandbox, you cannot access `localhost` directly on your physical machine.

Open a **new, separate local CMD window** on your computer and use the GitHub CLI to forward both your Game and Jenkins ports to your local machine:

```cmd
gh codespace ports forward 8080:8080 8082:8082 -c your-codespace-name

```

You can now open your web browser and access both services as if they were running locally:

* **Jenkins dashboard:** `http://localhost:8082`
* **Your Spring Game (once deployed):** `http://localhost:8080`

---

### Step 9: Configure Your Local SCM Pipeline

1. Open `http://localhost:8082` in your browser and paste your security key to unlock Jenkins.
2. Select **Install Suggested Plugins**.
3. Create your admin user, navigate to the Jenkins dashboard, and click **New Item**.
4. Name your pipeline `game-automation`, select **Pipeline**, and click **OK**.
5. Scroll down to the **Pipeline** configuration section, paste this automated script, and click **Save**:

```groovy
pipeline {
    agent any
    stages {
        stage('Pull SCM') {
            steps {
                // Jenkins pulls directly from your Codespace workspace path
                checkout scm
            }
        }
        stage('Deploy New Build') {
            steps {
                // Pull down previous game instances safely to free up port 8080
                sh 'docker stop game-instance || true'
                sh 'docker rm game-instance || true'
                
                // Build and start the game engine container
                sh 'docker build -t game-engine:latest .'
                sh 'docker run -d --name game-instance -p 8080:8080 game-engine:latest'
            }
        }
    }
}

```

---

### Step 10: Trigger the Build and Verify

Click **Build Now** on the left menu. Jenkins will pull the code from your Codespace folder, compile it, wrap it in your target Docker environment, and deploy it inside the cloud container.

Navigate to `http://localhost:8080` in your web browser to play your programming game!

There are **two key errors** in your Dockerfile that will prevent it from building successfully. Let's break them down so you can get this running smoothly.

---

### 1. The Critical Error: Invalid Maven Tag (Line 2)

```dockerfile
FROM maven:eclipse-temurin:17-jre-slim AS build

```

* **What is wrong:** You have stacked multiple colons in the image name (`maven:eclipse-temurin:17...`), which is invalid Docker syntax. Additionally, the official Maven registry does not have a `jre-slim` variation for Eclipse Temurin.
* **The fix:** Use a valid, official tag from Docker Hub.
* If you want Java 17 with Temurin: **`maven:3.9.6-eclipse-temurin-17`** (using JDK for compiling, which is required).



### 2. The Semantic Error: Mismatched Stage Names (Lines 2 & 11)

* **What is wrong:** In **Stage 1**, you defined the build environment as `AS build` (lowercase):
```dockerfile
FROM ... AS build

```


But in **Stage 2**, you are trying to copy the compiled JAR from a stage named `build` in lowercase, which is correct:
```dockerfile
COPY --from=build /app/target/*.jar game-app.jar

```


*Note: Make sure your stage naming matches exactly. If you use `AS build`, use `--from=build`.*

---

### 🛠️ The Corrected Dockerfile

Here is the fully fixed, production-ready version of your Dockerfile:

```dockerfile
# Stage 1: Build compilation environment (Needs a JDK to compile Java code)
FROM maven:3.9.6-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Clean, ultra-lightweight execution engine container (Runs the JRE)
FROM eclipse-temurin:17-jre
WORKDIR /app
# Grabs the compiled JAR from the 'build' stage
COPY --from=build /app/target/*.jar game-app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "game-app.jar"]

```

### 💡 Quick Tips

* **`-DskipTests`:** I added this to the `mvn clean package` step. When building a container image, running tests can slow down the build or fail if they depend on external resources (like databases) that aren't available during the build stage.
* **JDK vs JRE:** In Stage 1, we use a **JDK** image (automatically included in the `maven` image) because you need the compiler (`javac`) to package the application. In Stage 2, we switch to a minimal **JRE** (`eclipse-temurin:17-jre`) because you only need the lightweight runtime to execute the pre-built `.jar` file.

The error `invalid reference format` happens because you have **backslashes (`\`) with spaces around them** inside your command, or because you copied the backslashes while executing everything as a single line in your command prompt.

When using a single-line command prompt, backslashes are either not needed to break up lines (like in Windows CMD) or they get confused if there is a space right after them (like in Linux/bash).

---

### 🛠️ The Fix

Run the command in **one straight line** without any backslashes. This is the safest way to prevent formatting issues:

```bash
docker run -d -p 8082:8080 -p 50000:50000 -v jenkins_home:/var/jenkins_home -v /var/run/docker.sock:/var/run/docker.sock --name jenkins-master jenkins/jenkins:lts

```

---

### 💡 Why did this happen?

In Linux terminals, the backslash `\` is used to tell the shell *"I'm going to continue this command on the next line."*

If you copy-paste the whole block together or if there is even a single invisible **space character** typed directly after a backslash (like `\ ` instead of `\`), the terminal fails to parse the string and throws `invalid reference format`. Removing the backslashes entirely and running it as a single line clears up the formatting!
